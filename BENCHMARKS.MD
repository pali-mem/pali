## Philosophy

Pali benchmarks are fully self-contained and reproducible:

- **Synthetic fixtures** generated by local Ollama models (real model outputs, no template synthesis path).
- **Shell harnesses** for API-level workloads (`scripts/benchmark.sh`, `scripts/retrieval_quality.sh`).
- **Per-provider comparisons** (default `ollama`, optional `onnx`) on the same fixture.

The goal is not to publish impressive numbers. It's to give anyone cloning this repo a reproducible way to verify behavior on their own hardware.

## Reproducing Benchmarks

### Prerequisites

```bash
go run ./cmd/setup     # checks Ollama + config; ONNX model/runtime are optional advanced path
scripts/bench_setup.sh # pulls required models (see below)
```

**Models pulled by `bench_setup.sh`:**

| Model | Size | Role |
|---|---|---|
| `all-minilm` | ~45 MB | Default Ollama embedding model for benchmark/retrieval runs |
| `phi4-mini` | ~2.5 GB | Fixture generation model |

### Step 1 — Generate fixtures

```bash
scripts/gen_fixtures.sh --model phi4-mini --count 1000 --tenants 100 --parallel 8 --out test/fixtures/memories_real.json
```

This calls Ollama locally and produces a fixture JSON in the format Pali's store API expects.

### Step 2 — Run benchmarks

```bash
scripts/benchmark.sh --fixture test/fixtures/memories_real.json --backend sqlite --embedding-provider ollama --embedding-model all-minilm
scripts/retrieval_quality.sh --fixture test/fixtures/memories_real.json --top-k 10 --max-queries 200 --embedding-provider ollama --embedding-model all-minilm
```

Current script support:
- `sqlite`: implemented and measured end-to-end (tenant create + store + search).
- `qdrant` / `pgvector`: reserved for future backend wiring in API router.
- Retrieval quality harness: implemented for `/v1/memory/search` with `Top1HitRate`, `Recall@k`, `nDCG@k`, `MRR`, and `HitRate@k`.

### Step 3 — Read results

Results are written to `test/benchmarks/results/` as JSON and a human-readable summary. The summary format is what appears in the [Results](#results) section below.

Useful flags:
- `--search-ops <n>` controls number of timed search requests (default `200`)
- `--top-k <n>` controls search `top_k` (default `10`)
- `--base-url <url>` benchmarks an already-running server instead of starting one
- `--embedding-provider <ollama|onnx|mock>` controls embedder used by auto-started benchmark server
- `--embedding-model <name>` controls Ollama embedding model (default `all-minilm`)
- `--eval-set <path>` (retrieval quality script) uses labeled query cases; if omitted, cases are auto-generated from fixture rows and grouped by `tenant_id + query` (multi-relevant labels).
- `--sample-seed <n>` deterministic seed used when `--max-queries` samples a subset of eval cases.
- `scripts/retrieval_trend.sh` wraps retrieval runs and appends a JSONL history row per run for change-by-change tracking.
- `scripts/benchmark.sh` auto-detects `POST /v1/memory/batch`; when available, store writes use batched ingestion and results include `store.mode` (`batch` or `single`).

### Retrieval quality eval-set format

`scripts/retrieval_quality.sh` accepts an optional labeled eval-set JSON:

```json
[
  {
    "tenant_id": "bench_tenant_001",
    "query": "dark mode preferences",
    "expected_fixture_indexes": [0, 5]
  }
]
```

- `expected_fixture_indexes` are resolved to runtime memory IDs after store.
- `expected_memory_ids` can also be provided directly.
- Example files:
  - `test/fixtures/retrieval_eval.sample.json` (small smoke set)
  - `test/fixtures/retrieval_eval.curated.json` (larger curated set)

### Retrieval trend tracking

Track retrieval metrics per change with:

```bash
scripts/retrieval_trend.sh \
  --label "curated-baseline" \
  --fixture test/fixtures/memories.json \
  --eval-set test/fixtures/retrieval_eval.curated.json \
  --top-k 10 --max-queries 0 \
  --embedding-provider ollama --embedding-model all-minilm
```

Trend history file:
- `test/benchmarks/trends/retrieval_quality_history.jsonl`

## Fixture Generation

### Why synthetic

Benchmarks use synthetically generated memories because:

1. The same local models Pali is designed to serve also generate the test data — the loop is closed and self-consistent.
2. No dependency on external datasets or internet access.
3. Fixture distribution (categories, lengths, tenant spread) is fully controlled and documented.
4. Anyone can regenerate fixtures on their hardware with one command.

### What `gen_fixtures.sh` does

The script calls a locally running Ollama instance via `cohesion-org/deepseek-go` and generates memories across four categories, spread across a configurable number of synthetic tenants:

| Category | Description | Example |
|---|---|---|
| **Preferences** | Stable user preferences | `"User prefers dark mode and compact UI layouts."` |
| **Profile facts** | Durable personal/professional facts | `"User is a backend engineer working primarily in Go."` |
| **Episodic events** | One-off events, plans, session context | `"User mentioned they have a design review on Thursday."` |
| **Transient context** | Short-lived, session-specific constraints | `"User is working on a deadline and wants concise responses."` |

Distribution across categories is weighted to reflect realistic usage (more episodic + preferences, fewer profile facts). Length is varied — short (1 sentence), medium (2–3 sentences), long (4–5 sentences) — so the corpus is not artificially uniform.

### Fixture schema

Generated memories conform to Pali's store API format:

```json
[
  {
    "tenant_id": "bench_tenant_001",
    "content": "User prefers dark mode and compact UI layouts.",
    "tags": ["preferences"],
    "tier": "semantic"
  },
  {
    "tenant_id": "bench_tenant_001",
    "content": "User mentioned they have a design review on Thursday.",
    "tags": ["event"],
    "tier": "episodic"
  }
]
```

### Fixture sizes

Three fixture sizes are used to surface scaling behavior:

| Size | Memory count | Tenants | Purpose |
|---|---|---|---|
| `small` | 1,000 | 10 | Baseline; expected to be fast on all backends |
| `medium` | 10,000 | 100 | Realistic single-user or small-team deployment |
| `large` | 50,000 | 500 | Where sqlite-vec KNN degrades vs ANN-indexed backends |

## What We Test

### 1. Store latency

Time from API call to memory persisted (embedded + scored + written).

**Variants:**
- Heuristic scorer (default)
- Ollama scorer (opt-in) — expected to add 100–500ms per store

**Reported:** p50 / p95 / p99 over N=1000 store operations.

---

### 2. Search latency (two-phase retrieval)

Time from search query to ranked results returned.

**Breakdown:**
- Phase 1: vector candidate retrieval (K'=50, 100, 200)
- Phase 2: WMR scoring over candidate set
- Total end-to-end

**Variants:** run against small / medium / large fixture sizes, per backend.

**Reported:** p50 / p95 / p99 per phase and total.

---

### 3. Embedding latency

Time to embed a single input string via MiniLM-L6-v2 / ONNX Runtime.

Isolated from store/search so ONNX performance is independently visible.

**Reported:** p50 / p95 / p99 over N=1000 inputs of varying length.

---

### 4. Backend scaling comparison

Search latency at small / medium / large fixture sizes, across all three backends.

This is the most important benchmark — it directly answers "when should I switch from sqlite-vec to Qdrant?"

**Reported:** p99 search latency per backend per fixture size (summary table).

---

### 5. Concurrent write throughput (SQLite WAL)

Simulates N concurrent tenants writing memories simultaneously to the sqlite-vec backend.

This surfaces the WAL contention point and validates the recommendation to switch backends under heavy concurrent write load.

**Reported:** total throughput (writes/sec) at N=1, 5, 10, 25, 50 concurrent writers.

## Fixture Generation — Implementation Notes for Contributors

> This section is for whoever implements `scripts/gen_fixtures.sh` and `cmd/genfix/`.

### Script entrypoint

`scripts/gen_fixtures.sh` is a thin shell wrapper. The actual generation logic lives in `cmd/genfix/main.go`.

```
cmd/
└── genfix/
    └── main.go    # fixture generation binary
```

### Generation logic

The generator should:

1. Accept `--model`, `--count`, `--tenants`, `--out`, `--seed`, and `--parallel` flags.
2. Call the local Ollama instance and fail fast with a clear setup error when unavailable.
3. For each memory, prompt the model to generate a realistic memory string for a given category. Keep prompts short and deterministic — the goal is variety, not creativity.
4. Assign memories to tenants with deterministic seeded randomness.
5. Set `tier` based on category: preferences/profile → `semantic`, episodic/transient → `episodic`.
6. Write output to `--out` as a JSON array matching the store API schema above.
7. Print a generation summary: total count, per-category breakdown, time taken, and throughput.

### Prompt template (starting point)

```text
Generate a single realistic memory for an AI assistant to store about a user.
Category: {category}
Length: {short|medium|long}
Return only the memory string. No preamble, no explanation.
```

Adjust and iterate — the quality of generated memories directly affects how meaningful the benchmark results are.

### Determinism

Pass `--seed` through to the Ollama request (if the model supports it) so fixture generation is reproducible across runs. Check this into `test/fixtures/README.md` alongside the seed and model used.
